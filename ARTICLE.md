## Анотація

У статті проведено порівняльний аналіз чотирьох активно підтримуваних інструментів рендерингу API-документації у форматі Apiary 1A (API Blueprint): **Blueprinter**, **Aglio**, **Api-blueprint-boilerplate** та **Phpdraft**. Дослідження зумовлене необхідністю міграції документації Єдиної системи охорони здоров'я України (ЕСОЗ) у зв'язку із завершенням строку підтримки (End-of-Life, EoL) платформи Oracle Apiary. На підставі формалізованого набору критеріїв — повноти підтримки MSON, якості обробки попереджень парсера, зручності УІ, автономності від хмарних сервісів Oracle та придатності до роботи з масштабними специфікаціями — обґрунтовано вибір **Blueprinter** як оптимального інструменту для безпомилкової міграції. Практичним артефактом дослідження є репозиторій `ehealth-ua/ONE-API` та мігрований файл `esoz-blueprint.apib`.

**Ключові слова:** API Blueprint, APIB, Oracle Apiary, ЕСОЗ, міграція документації, Blueprinter, Aglio, MSON, рендерер, сумісність.

---

## Abstract

This paper presents a comparative analysis of four actively maintained API Blueprint (Apiary 1A format) documentation rendering tools: Blueprinter, Aglio, Api-blueprint-boilerplate, and Phpdraft. The study is motivated by the necessity of migrating the Ukrainian Electronic Health System (eHealth) API documentation following the End-of-Life announcement for Oracle Apiary. Based on a formalized set of criteria — MSON support completeness, parser warning handling quality, UI ergonomics, independence from Oracle cloud services, and scalability for large specifications — **Blueprinter** is justified as the optimal migration target. The practical artifact of this research is the `ehealth-ua/ONE-API` repository and the migrated `esoz-blueprint.apib` specification file.

**Keywords:** API Blueprint, APIB, Oracle Apiary, eHealth Ukraine, documentation migration, Blueprinter, Aglio, MSON, renderer, compatibility.

## 1. Вступ

Документація програмних інтерфейсів (API) є стратегічним компонентом архітектури сучасних державних інформаційних систем. У рамках цифрової трансформації охорони здоров'я України функціонує Єдина система охорони здоров'я (ЕСОЗ) — централізована інформаційна платформа, що інтегрує медичні інформаційні системи (МІС), лабораторії, аптеки та регуляторні органи. Взаємодія між учасниками екосистеми здійснюється через REST API, специфікації якого традиційно описуються у форматі **API Blueprint** і публікуються через платформу **Oracle Apiary**.

Проте Oracle оголосила про завершення активної підтримки платформи Apiary (End-of-Life), що ставить перед командою ЕСОЗ завдання вибору альтернативного інструментарію. Критичним є збереження можливості рендерингу існуючих специфікацій формату APIB без значних переробок, оскільки основний файл `apiary.apib` має об'єм понад 1 МБ і охоплює сотні ресурсів, типів даних MSON та унікальних розширень формату.

**Мета статті** — провести систематичний порівняльний аналіз чотирьох альтернативних інструментів та науково обґрунтувати доцільність вибору Blueprinter як основного рендерера для документації ЕСОЗ.

## 2. Огляд формату API Blueprint та його інструментальної екосистеми

API Blueprint — це мова опису веб-API рівня документації, розроблена компанією Apiary (Jakub Nešetřil та Zdenko Vrábel, 2013). Вона ґрунтується на синтаксисі Markdown та MSON (Markdown Syntax for Object Notation) і зберігається у файлах з розширенням `.apib` [1]. Специфікація підтримується відкритою бібліотекою `drafter` (C++) та парсером `minim` (JavaScript), які реалізують перетворення тексту у дерево API Elements (IETF Internet Draft).

ЕСОЗ з самого початку побудована навколо цього формату. Всі специфікації REST-ресурсів — від `Patient`, `Encounter`, `Composition` до `MedicationRequest` — описані у `.apib` файлах та зводяться в єдину специфікацію `apiary.apib` обсягом понад мільйон байт.

Ключовою відмінністю між доступними рендерерами є глибина їхньої інтеграції з виходом парсера: деякі з них працюють лише з AST-представленням першого покоління (`apib-ast`), тоді як сучасніші — з API Elements другого покоління, що є більш семантично насиченим.

## 3. Формальний аналіз порівнюваних інструментів

### 3.1. Aglio

**Автор та генеалогія.** Daniel G. Taylor (GitHub: `danielgtaylor`). Проєкт `danielgtaylor/aglio` — один із перших і найбільш відомих рендерерів API Blueprint. Початково презентований близько 2014 року, він дав поштовх розвитку всієї екосистеми відкритих рендерерів APIB. Має понад 4 000 зірок на GitHub.

**Технічні характеристики.** Написаний на Node.js.架構розподілена: ядро (core) завантажує `.apib`-файл та обробляє аргументи, а тематичний рушій (theme engine) конвертує AST у HTML. За замовчуванням використовує тему Olio; підтримує також Streak, Flatly, Slate, Cyborg. Підтримує вбудовані директиви `<!-- include -->` для поєднання кількох APIB-файлів.

**Сильні сторони:**
- Велика база встановлень та документації.
- Підтримка кастомних Jade-шаблонів.
- Вбудований live-preview сервер.
- Широка інтеграція у CI/CD пайплайни (Docker-образи, плагіни для Foliant, mkdocs тощо).

**Слабкі сторони:**
- З 2019 року автор офіційно повідомив про обмежену підтримку та запросив нових мейнтейнерів [2].
- Залежить від застарілих версій `drafter`/`protagonist`, що обумовлює некоректну обробку новітніх розширень MSON (зокрема, вкладених членів підтипів примітивних типів), генеруючи численні попередження при рендерингу специфікацій ЕСОЗ.
- Відсутність нативного UX для мобільних пристроїв та темної теми.

---

### 3.2. Api-blueprint-boilerplate

**Автор та генеалогія.** Jakub Synowiec (GitHub: `jsynowiec`), проєкт `jsynowiec/api-blueprint-boilerplate`. Не є самостійним рендерером; являє собою готовий starter-kit (шаблон репозиторію) для швидкого старту проєктів документації у форматі APIB.

**Технічні характеристики.** Включає визначений набір залежностей: `apiary-cli` для локального перегляду та валідації (використовує хмарний API Apiary), `drafter.js` для тестування схеми, `aglio` для генерації статичного HTML. Вимагає наявності Node.js, NPM, Ruby та Bundler.

**Сильні сторони:**
- Готова структура проєкту з прикладами, CI-конфігурацією та документацією.
- Вбудований `npm test` для швидкої валідації схеми через drafter.js.

**Слабкі сторони:**
- Принципово залежить від `apiary-cli` — компонента екосистеми Oracle Apiary, що підлягає міграції. Після EoL Apiary інструмент у базовій конфігурації стає непридатним.
- На відміну від інших кандидатів, не є рендерером, а лише обгорткою, тому не може безпосередньо замінити Apiary у виробничому середовищі.
- Відсутня власна система тем та функції пошуку.

---

### 3.3. Phpdraft

**Автор та генеалогія.** Sean Miller (GitHub: `SMillerDev`), репозиторій `SMillerDev/phpdraft`. Первинний автор — Ryan Hellenius (попередня версія). Це єдиний PHP-реалізований парсер та рендерер API Blueprint у наведеному переліку.

**Технічні характеристики.** Написаний на PHP 8.1+. Для парсингу `.apib` потребує системного бінарного файлу `drafter` або може використовувати онлайн-парсер Apiary за відсутності локального. Підтримує генерацію HTML та конвертацію у формат OpenAPI (Swagger).

**Сильні сторони:**
- Можливість нативної інтеграції в PHP-додатки (Laravel, Symfony).
- Функція конвертації APIB → OpenAPI — цінна при стратегії повноцінної міграції на OpenAPI у майбутньому.
- Підтримка CSS та JavaScript-кастомізації шаблонів.

**Слабкі сторони:**
- Потребує налаштування окремого середовища (PHP + Drafter), що ускладнює DevOps для команд, що використовують Node.js.
- Залежність від системного `drafter` робить інструмент чутливим до версій бінарного файлу.
- Менш ергономічний UX при роботі з масштабними специфікаціями: відсутність вбудованого пошуку та навігаційних фільтрів.

---

### 3.4. Blueprinter

**Автор та генеалогія.** Команда Funbox (GitHub: `@funboxteam`), репозиторій `funboxteam/blueprinter`. Розроблений як сучасна альтернатива Aglio з акцентом на глибоку інтеграцію з API Elements другого покоління.

**Технічні характеристики.** Написаний на Node.js. Встановлюється через npm як `@funboxteam/blueprinter`. Приймає `.apib`-файл та генерує самодостатній HTML-файл (`index.html`) без зовнішніх залежностей. Підтримує режим watch/live-server.

**Сильні сторони:**
- **Глибока інтеграція з API Elements:** використовує сучасну версію `minim` та `drafter`, що забезпечує коректний парсинг новітніх конструкцій MSON.
- **Детальна обробка помилок парсера:** чітке відображення попереджень безпосередньо в HTMLінтерфейсі з прив'язкою до рядків файлу.
- **Темна тема:** нативна підтримка без додаткових налаштувань.
- **Повнотекстовий пошук:** можливість пошуку по групах, ресурсах та діях.
- **Інтерактивні «One Of» блоки:** динамічна зміна варіантів відповіді без перезавантаження сторінки.
- **Версія для друку:** адаптований режим без зайвих елементів.
- **Локалізація UI:** можливість задати мову інтерфейсу.
- **Повна автономність:** не потребує сервісів Oracle; працює локально або в CI/CD.

**Слабкі сторони:**
- Менша спільнота порівняно з Aglio.
- Відсутність підтримки директиви `<!-- include -->` для розбиття специфікації на файли (однак це особливість формату Bootstrap-файлів, а не критична вада).

## 4. Формалізація вимог до міграції

Виходячи з профілю ЕСОЗ (масштабна APIB-специфікація, екосистема Node.js, необхідність автономності, велика стороння аудиторія МІС-розробників), сформовано такі критерії відбору:

| № | Критерій | Обґрунтування |

| K2 | **Мінімальна кількість попереджень/помилок при рендерингу** | Безпосередній метрик якості міграції |
| K3 | **Автономність від сервісів Oracle Apiary** | Ключова передумова задачі (EoL) |
| K4 | **Пошук та навігація в UX** | МІС-розробники потребують зручного доступу до специфікацій |
| K5 | **Підтримка середовища Node.js** | Відповідність поточному технологічному стеку команди |
| K6 | **Активна підтримка та оновлення** | Зниження ризику повторного EoL у середньостроковій перспективі |

## 6. Практична апробація: міграція `apiary.apib` → `esoz-blueprint.apib`

Для верифікації теоретичних висновків було проведено практичну міграцію у репозиторії `ehealth-ua/ONE-API`. Оригінальний файл `apiary.apib` (1 061 617 байт) було адаптовано до вимог Blueprinter та збережено як `esoz-blueprint.apib` (1 007 359 байт). Серед виконаних коригувань:

1. **Усунення конструкцій `sub-types of primitive types should not have nested members`** — системне попередження, яке Blueprinter відображає явно, а Aglio ігнорує, проте генерує некоректний HTML.
2. **Виправлення порожніх `Enum` секцій** — ідентифіковано та виправлено помилки `Enum element should include members`, що призводили до збоїв рендерингу.
3. **Нормалізація URI-шаблонів** — усунення дублікатів та неканонічних параметрів, що спричиняли попередження парсера.
4. **Виправлення посилань `Include`** — видалено некоректні `Include`-директиви в нетипових позиціях.

Результуючий файл `esoz-blueprint.apib` успішно рендериться командою:

```bash
blueprinter -i esoz-blueprint.apib -o index.html
```

без критичних помилок та з мінімальною кількістю несуттєвих попереджень. Генерований `index.html` (10 716 937 байт) є автодостатнім документом, доступним для перегляду у будь-якому сучасному браузері без підключення до мережі.

Для зручності розробки у репозиторії визначено npm-скрипти:

```json
{
  "scripts": {
    "dev": "blueprinter -i apiary.apib -s -p 3000",
    "doc": "blueprinter -i apiary.apib -o index.html"
  },
  "dependencies": {
    "@funboxteam/blueprinter": "^6.1.0"
  }
}
```

Ці скрипти забезпечують як розробницький режим з hot-reload, так і генерацію фінального артефакту документації.

## 7. Висновки

За результатами дослідження встановлено:

1. **Aglio**, незважаючи на широку популяризованість, перебуває у стані обмеженої підтримки та має системні проблеми з парсингом сучасних конструкцій MSON, що критично важливо для ЕСОЗ.

2. **Api-blueprint-boilerplate** є стартовим шаблоном, а не повноцінним продуктом, і принципово залежить від хмарних сервісів Oracle Apiary, що унеможливлює його розгляд як цільової платформи після EoL.

3. **Phpdraft** є функціонально придатним варіантом, однак потребує PHP-інфраструктури та не забезпечує належного UX для широкої аудиторії МІС-розробників.

4. **Blueprinter** демонструє найвищу повноту підтримки формату API Blueprint 1A, мінімальну кількість помилок при рендерингу, сучасний та ергономічний UI, повну автономність від Oracle Apiary та відповідність технологічному стеку команди ЕСОЗ.

Практична апробація підтвердила архітектурну правильність вибору: репозиторій `ehealth-ua/ONE-API` з мігрованим файлом `esoz-blueprint.apib` є відтворюваним науковим артефактом, що документує весь процес міграції. Blueprinter рекомендується як стандартний інструмент рендерингу API-документації ЕСОЗ на заміну Oracle Apiary.
